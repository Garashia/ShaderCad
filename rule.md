# OSS管理・GitHub公開に関するルールと注意点

本プロジェクト（ShaderCad）において、通常NuGetなどで管理されるライブラリ群をGitサブモジュール等で直接ソース管理・利用し、かつGitHubに公開するにあたっての注意事項と運用ルールを定めます。

## 1. 依存ライブラリの管理方法と注意点

*   **NuGetではなくソースを取り込む理由**: 
    開発者がライブラリの内部動作を確認・デバッグしやすくするため、また特定のバージョンや修正をローカルで維持しやすくするためです。
*   **Gitサブモジュールの運用**: 
    `libs/` フォルダ以下に各OSSリポジトリをGitサブモジュールとして配置します。
    *   **注意点**: GitHubから本リポジトリをクローンするユーザーは、`git clone --recursive` を実行するか、クローン後に `git submodule update --init --recursive` を実行する必要があります。この手順を `README.md` に必ず明記してください。
    *   **更新の管理**: サブモジュール内のコミットポインタがずれると、ビルドできなくなる可能性があります。依存ライブラリのバージョンを上げる際以外は、サブモジュール内のコミットを不用意に変更しないようにしてください。

## 2. ライセンスの遵守と明記

商用利用やオープンソース化において、各OSSのライセンスを遵守する必要があります。

*   **MIT / BSD / Boost Software License**:
    *   対象: Avalonia UI (MIT), Silk.NET (MIT), Tomlyn (BSD-2-Clause), Geometry3Sharp (Boost 1.0)
    *   **ルール**: 非常に寛容なライセンスであり、ソースコードの同梱や公開、商用利用に問題はありません。ただし、著作権表示およびライセンス条文の記載義務があるため、プロジェクトの `NOTICE` ファイルや `README.md`（または各ライブラリの元の `LICENSE` ファイル）を保持・明記してください。
*   **LGPL 2.1**:
    *   対象: OpenCASCADE (フェーズ2での参考・利用)
    *   **ルール**: LGPLのコードを静的リンクすると、プロジェクト全体をLGPLで公開する義務が生じます。これを避けるため、OpenCASCADEは必ず **動的リンク（DLLとしての利用）** または P/Invoke を介した利用にとどめてください。
    *   動的リンクであれば、ShaderCad本体をMIT等の別のライセンスで公開することに問題はありません。ただし、「LGPLのライブラリを使用していることの明記」と、「ユーザーがLGPL部分のライブラリ（DLL）を差し替え可能にすること」が必要です。

## 3. リポジトリの肥大化防止 (OpenCASCADE等)

*   OpenCASCADEなどの非常に巨大なリポジトリは、サブモジュールとして追加するだけでもクローン時間やディスク容量を著しく圧迫します。
*   **ルール**: 「参考程度」や「巨大なアセット」を含むリポジトリは、メインリポジトリのサブモジュールにはせず、必要に応じてローカルの別ディレクトリでクローンして参照するか、ビルド済みのバイナリ（DLL）のみをプロジェクトに含める（または実行時にダウンロードする）運用を検討してください。
